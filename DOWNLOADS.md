# Signed downloads — Phase 2A

This document describes the plug-in download pipeline. Together with
`WAITLIST.md` (Phase 1) and `ADMIN.md` (Phase 2B) it covers every
operational surface of the website.

## Goal

A confirmed waitlist subscriber visits a plug-in page, enters their
email, and within a minute receives an email containing **signed,
24h-valid download URLs** for every currently-shipping platform of
that plug-in.

The mechanism is the same for the entire Collection (each link is
generated per `<slug>/<platform>/<version>` triplet) and for a single
plug-in. The user never sees S3-style URLs, signatures, or storage
internals.

## End-to-end flow

```
┌────────────────────────────────────────────────────────────────────┐
│ Browser — opticksaudio.com/plugins/reflexion                       │
│   1. Click "Download for Mac" (or "Get my download links")         │
│   2. Inline DownloadsForm reveals                                  │
│   3. POST /api/downloads/request                                   │
│         body: { email, plugins?, source }                          │
└────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌────────────────────────────────────────────────────────────────────┐
│ Next.js route handler — src/app/api/downloads/request/route.ts     │
│   - Zod-validates body, rate-limits per-IP (3/min), honeypots      │
│   - Looks up subscriber. If not 'confirmed' → silent OK            │
│     (anti-enumeration; no list-membership leak)                    │
│   - Reads plugin_releases where is_current = true                  │
│   - For each {slug, platform}:                                     │
│       HEAD the R2 object → only sign URLs that actually exist      │
│   - Inserts download_grants row (audit trail) with version_pins    │
│   - Calls sendDownloadEmail(...) via Resend                        │
└────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌────────────────────────────────────────────────────────────────────┐
│ User's inbox                                                       │
│   Email subject: "Your Opticks Collection is ready to download"    │
│   Per-plug-in card with platform-specific download buttons         │
│   macOS Gatekeeper instructions (right-click → Open → Open again)  │
└────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌────────────────────────────────────────────────────────────────────┐
│ Click "Download for Mac" → R2 signed URL                           │
│   - Validity: 24 hours from issue                                  │
│   - Bound to a specific Bucket + Key                               │
│   - Content-Disposition forces a clean filename                    │
└────────────────────────────────────────────────────────────────────┘
```

## Storage layout in R2

Bucket: **`opticks-audio-releases`** (private; no public read).

```
opticks-audio-releases/
├── reflexion/
│   └── v0.1.0/
│       └── Reflexion-mac-v0.1.0.zip
├── refraction/
│   └── v0.1.0/
│       └── Refraction-mac-v0.1.0.zip
└── inflexion/
    └── v0.1.0/
        └── Inflexion-mac-v0.1.0.zip
```

The key format is generated by `pluginObjectKey()` in
`src/lib/r2.ts`:

```ts
`${slug}/v${version}/${Capitalised(slug)}-${platform}-v${version}.zip`
```

When Windows builds arrive, the same layout is used with `-windows-`
in place of `-mac-`, and the `plugin_releases.platforms` array is
updated to include `'windows'`.

## Database tables involved

### `plugin_releases`
The catalog of what versions exist. The website reads
`is_current = true`. Old versions stay in the table forever; they are
just no longer offered to new requesters.

| column | purpose |
|---|---|
| `slug` | one of `reflexion`, `refraction`, `inflexion` |
| `version` | semver-ish text (`0.1.0`, `0.2.0-beta.1`, etc.) |
| `platforms` | text[] declaring what's actually uploaded — `{mac}`, `{mac,windows}` |
| `release_notes` | a sentence used in the download email |
| `is_current` | only one current row per slug |

### `download_grants`
One row per "user asked for download links" event. Holds the version
pin so re-issuing the same grant later gives identical files.

### `download_events`
Reserved for Phase 2E — a redirect endpoint will wrap each signed URL
and record per-click telemetry (slug, platform, version, IP hash, UA,
result). Empty today.

## Producing release artefacts

There is a packaging script that builds the per-plug-in `.zip`
archives expected by R2:

```bash
bash scripts/package-release.sh 0.1.0
# Output: ~/Downloads/opticks-releases/v0.1.0/
#   Reflexion-mac-v0.1.0.zip
#   Refraction-mac-v0.1.0.zip
#   Inflexion-mac-v0.1.0.zip
```

Inputs the script reads:
- `~/Library/Audio/Plug-Ins/VST3/{Reflexion,Refraction,Inflexion}.vst3`
- `~/Library/Audio/Plug-Ins/Components/{...}.component` (optional —
  bundled if present)

Each archive contains the platform binaries plus a `README.txt` with
install instructions and the Gatekeeper workaround.

To publish a new version, follow the four-step recipe in
`CLAUDE.md` → "Release a new plug-in version".

## Failure modes and how the system reacts

| Situation | Response |
|---|---|
| Email not in `subscribers` | `{ok:true,status:queued}` (silent) |
| Email is `pending` (not confirmed) | `{ok:true,status:queued}` (silent) |
| `plugin_releases` has no current row | `503 no_release` |
| R2 object missing for the requested platform | URL skipped; email still sent with whatever is available |
| All requested R2 objects missing | `503 no_artefacts` |
| Resend delivery throws | `{ok:true,status:queued,warn:"email_delivery_delayed"}` and the `download_grants` row is kept so an admin can replay manually |
| Rate limit exceeded (3/min/IP) | `429 rate_limited` |
| Honeypot field filled | `{ok:true,status:queued}` (silent) |

## Why HEAD before sign

Signing a URL is free and works whether or not the object exists.
Emailing a signed URL that 404s would be a brand-damaging dead-end for
the customer. The R2 client therefore **HEADs every key before
signing**, and the email is built only from URLs that resolved.

If the HEAD costs become a concern, switch to a cache keyed by
`(slug, version)` with a short TTL — the catalog rarely changes.
